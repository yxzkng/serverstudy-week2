# 03 — 실습 (뒷타임)

> **목표는 명령어 암기가 아니다.** 서버를 이미지로 묶은 뒤, 실행 상태·로그·포트·자원·재시작을
> 어떻게 관리하는지 **직접 확인**하는 것이다. 각 단계 끝에 있는 ✅ **확인 포인트**를 눈으로 보고 넘어가자.
>
> 명령은 **Windows(PowerShell)** 기준. macOS/Linux는 `.\gradlew.bat` → `./gradlew`로 바꾼다.

---

## 사전 조건

- `docs/01-환경세팅.md`를 끝냈다 (Docker Desktop 실행 중 + `spring-petclinic:1.0` 이미지 존재)
- spring-petclinic 폴더 안에서 시작한다

빠르게 재확인:

```powershell
docker info                      # daemon 정상?
docker images | findstr petclinic  # spring-petclinic:1.0 있나?
```

## STEP 1. Docker가 준비됐는지 확인

```powershell
docker --version
docker info
```

✅ **확인 포인트**: 버전이 출력되고, `docker info`가 오류 없이 데몬 정보를 보여준다.
안 되면 → Docker Desktop이 안 켜진 것. 먼저 실행한다.

---

## STEP 2. PetClinic 이미지 만들기

(01에서 미리 빌드했다면 건너뛰어도 된다. 안 했다면 지금 한다.)

```powershell
.\gradlew.bat bootBuildImage --imageName=spring-petclinic:1.0
```

빌드가 끝나면:

```powershell
docker images | findstr petclinic
```

✅ **확인 포인트**: `spring-petclinic   1.0   <IMAGE ID>   ...   <크기>` 한 줄이 보인다.

> 🧠 생각해보기: 이 이미지 안에는 우리가 자바를 따로 설치하지 않아도 앱이 돌 수 있게 **자바 런타임까지** 들어 있다. 크기가 몇백 MB인 이유다.

---

## STEP 3. 재시작 정책 + 자원 제한을 걸고 실행

이번 실습의 핵심 한 줄이다.

```powershell
docker run -d --name petclinic-app --restart unless-stopped --memory 1g --cpus 1 -p 8081:8080 spring-petclinic:1.0
```

옵션 복습:

| 옵션 | 뜻 |
|---|---|
| `-d` | 백그라운드 실행 (터미널 닫아도 계속 돎) |
| `--name petclinic-app` | 컨테이너 이름 |
| `--restart unless-stopped` | 죽으면 자동 재시작 (내가 stop한 건 제외) |
| `--memory 1g` `--cpus 1` | 자원 상한 |
| `-p 8081:8080` | 내 8081 → 컨테이너 8080 |

✅ **확인 포인트**: 긴 컨테이너 ID 문자열이 출력되면 실행 성공.

```powershell
docker ps
```

`STATUS`가 `Up ...` 이고 `PORTS`에 `0.0.0.0:8081->8080/tcp`가 보이면 정상이다.

> ⚠️ 방금 떴어도 자바 앱은 **부팅 중**일 수 있다. STEP 4에서 헬스체크로 "앱까지" 준비됐는지 본다.

---

## STEP 4. 브라우저에서 서버 확인

부팅에 20~40초 걸릴 수 있다. 잠시 기다렸다가 접속한다.

- 앱 화면: http://localhost:8081
- 헬스체크: http://localhost:8081/actuator/health

✅ **확인 포인트 (2가지)**:
1. `http://localhost:8081` 에서 PetClinic 첫 화면이 뜬다.
2. `/actuator/health` 에서 `{"status":"UP"}` 이 보인다.

> 🧠 핵심 개념: `docker ps`가 `Up`이어도 방금 막 떴다면 헬스체크가 아직 `UP`이 아닐 수 있다.
> **컨테이너 Up(상자 실행) ≠ 앱 UP(요청받을 준비 완료).** 이 둘의 시차를 눈으로 확인하자.

### (선택) 로컬 서버와 동시에 띄우기

지난주처럼 로컬에서도 PetClinic을 띄워보자. 새 터미널에서:

```powershell
.\gradlew.bat bootRun
```

이건 8080으로 뜬다. 그럼 이제:
- http://localhost:8080 → **로컬** 서버 (컨테이너 아님)
- http://localhost:8081 → **컨테이너** 서버

✅ **확인 포인트**: 같은 앱이 포트만 다르게 **두 개** 동시에 떠 있다. 이게 포트 매핑의 의미다.

## STEP 5. 상태 · 로그 · 자원 사용량 확인

```powershell
docker ps
docker logs --tail 80 petclinic-app
docker stats petclinic-app
docker inspect petclinic-app
```

하나씩 보자.

- `docker ps` — 실행 중 컨테이너 목록. 죽은 것까지 보려면 `docker ps -a`.
- `docker logs --tail 80 petclinic-app` — 최근 로그 80줄. Spring 부팅 로그, `Started PetClinicApplication ...` 라인이 보이면 앱이 정상 기동한 것.
- `docker stats petclinic-app` — **실시간** CPU·메모리. `MEM USAGE / LIMIT`에서 우리가 건 `1GiB` 한도가 보인다. **다 봤으면 `Ctrl + C`로 빠져나온다.**
- `docker inspect petclinic-app` — 전체 설정(JSON, 매우 길다). 특정 값만:
  ```powershell
  docker inspect --format '{{.State.Status}}' petclinic-app
  docker inspect --format '{{.HostConfig.RestartPolicy.Name}}' petclinic-app
  ```

✅ **확인 포인트**:
- `logs`에서 `Started PetClinicApplication` 라인을 찾았다.
- `stats`에서 메모리 LIMIT이 `1GiB` 근처로 나온다.
- `inspect`에서 RestartPolicy가 `unless-stopped`로 나온다.

## STEP 6. 강제 종료 → 자동 재시작 확인

서버가 갑자기 죽는 상황을 흉내 낸다.

```powershell
docker kill petclinic-app
docker ps -a
```

`kill`은 **강제(비정상) 종료**라 재시작 정책이 발동한다. 잠깐 기다렸다가 다시:

```powershell
docker ps
```

✅ **확인 포인트**: 잠시 후 `petclinic-app`이 다시 `Up`으로 살아나 있다.
`STATUS`의 "Up X seconds"가 방금으로 리셋됐으면 재시작된 것이다.

> ⚠️ **함정**: `kill` 직후 바로 `docker ps`를 치면 이미 부활해서 "안 죽었네?"처럼 보인다.
> `docker ps -a`로 `Restarting`/`Exited`→`Up` 변화를 잡거나, 몇 초 뒤 STATUS 시간이 리셋된 걸로 확인한다.

### 비교 실험: stop은 다르다

```powershell
docker stop petclinic-app
docker ps          # 목록에 없다 (부활 안 함)
docker ps -a       # Exited 상태로 보인다
```

✅ **확인 포인트**: `stop`(정상 종료)은 `unless-stopped`에서 **부활하지 않는다**.
`kill`(강제)과의 차이를 확인했으면, 다시 켜둔다:

```powershell
docker start petclinic-app
```

## STEP 7. 이미지를 파일로 만들어 다른 컴퓨터에서 실행

```powershell
docker save -o spring-petclinic-1.0.tar spring-petclinic:1.0
```

현재 폴더에 `spring-petclinic-1.0.tar`가 생긴다(수백 MB). 이 파일을 짝꿍에게 넘긴다(USB·메신저 등).

**받은 사람** 컴퓨터에서:

```powershell
docker load -i spring-petclinic-1.0.tar
docker run --name petclinic-app -p 8081:8080 spring-petclinic:1.0
```

✅ **확인 포인트**: 소스코드도, `gradlew` 빌드도 없이 **tar 파일 하나만으로** 같은 서버가 http://localhost:8081 에 뜬다. 이것이 "이식성"이다.

> 짝을 지어 서로의 tar를 주고받아 실행해보면 확실히 체감된다.

### (선택) Docker Hub로 올리고 받기

시간이 남으면. 먼저 hub.docker.com 가입 + `docker login` 후:

```powershell
docker tag spring-petclinic:1.0 <내계정>/spring-petclinic:1.0
docker push <내계정>/spring-petclinic:1.0
```

다른 컴퓨터에서:

```powershell
docker pull <내계정>/spring-petclinic:1.0
```

## STEP 8. 정리 (실습 끝나고)

```powershell
docker stop petclinic-app         # 멈추고
docker rm petclinic-app           # 컨테이너 삭제
docker rmi spring-petclinic:1.0   # (원하면) 이미지도 삭제
del spring-petclinic-1.0.tar      # tar 파일 정리 (macOS/Linux는 rm)
```

로컬 `bootRun`을 켰다면 그 터미널에서 `Ctrl + C`로 종료한다.
