# 02 — 이론 (앞타임 발표 + 예습)

> 이번 주 실습을 이해하는 데 필요한 최소한의 개념만 담았다.
> 더 자세한 설명(비유 위주)은 노션 자료를 참고. 여기서는 **실습과 직접 연결되는 것**만 짚는다.

---

## 1. 왜 Docker인가 — "내 컴퓨터에선 되는데요" 문제

내 노트북에선 되는 서버가 팀원·배포 서버에선 안 뜨는 일이 흔하다. 자바 버전, 환경변수, OS가 달라서다.

Docker는 앱을 **컨테이너**라는 상자에 담는다. 이 상자엔 앱 + 실행에 필요한 모든 것(런타임·라이브러리·설정)이 들어 있어서, 받는 컴퓨터 환경과 무관하게 **똑같이** 돈다.

→ 오늘 실습의 `docker save` / `docker load`가 바로 이 "이식성"을 증명한다.

### 컨테이너 vs VM

| | VM | 컨테이너 |
|---|---|---|
| OS | 통째로 포함 | 호스트 OS 커널 **공유** |
| 크기 | GB | MB |
| 시작 | 느림 | 빠름 |

VM은 무겁고, 컨테이너는 가볍다. 컨테이너는 호스트 OS를 빌려 쓰기 때문이다.

## 2. Dockerfile · Image · Container

요리에 비유하면 한 번에 정리된다.

```
Dockerfile  --build-->  Image  --run-->  Container
  레시피                  밀키트            요리(실체)
```

- **Dockerfile**: 이미지를 어떻게 만들지 적은 설계도(레시피)
- **Image**: 실행에 필요한 걸 다 담은 **불변** 패키지(밀키트)
- **Container**: 이미지를 실제로 실행한 것(요리). 이미지 하나로 여러 개 찍어낼 수 있다.

### PetClinic엔 Dockerfile이 없다

PetClinic엔 Dockerfile이 없다. 대신 Spring Boot의 `bootBuildImage`가 **Cloud Native Buildpacks**로 이미지를 자동 생성한다. "레시피를 우리가 안 써도 Spring Boot가 대신 써준다"고 이해하면 된다.

```powershell
.\gradlew.bat bootBuildImage --imageName=spring-petclinic:1.0
```

`:1.0`은 **태그**(버전 이름표). 안 붙이면 `latest`가 된다.

## 3. 포트 매핑 — 이번 실습에서 제일 중요한 개념

컨테이너는 격리된 상자다. 안에서 8080으로 서버가 떠 있어도 밖에서 바로 접속 못 한다. `-p`로 상자에 **구멍을 뚫어 연결**한다.

```
-p 8081:8080
   호스트 : 컨테이너
```

- 읽는 법: "내 컴퓨터 8081로 온 요청을 컨테이너 안 8080으로 전달"
- 순서는 **호스트:컨테이너**. 뒤집으면 안 된다.
- 접속은 `http://localhost:8081` (컨테이너 안 8080이 아니라!)

**왜 8081?** 지난주 로컬 PetClinic이 8080을 쓸 수 있어서, 컨테이너를 8081로 빼면 **둘을 동시에** 띄워 비교할 수 있다.

## 4. 재시작 정책 (`--restart`)

서버는 버그·메모리 부족·재부팅 등으로 죽는다. 사람이 계속 지킬 수 없으니 자동으로 살아나게 한다.

| 정책 | 동작 |
|---|---|
| `no` | 재시작 안 함 (기본값) |
| `on-failure` | 오류로 죽을 때만 |
| `always` | 항상 (내가 stop해도 도커 재시작 시 부활) |
| `unless-stopped` | 항상, 단 **내가 직접 멈춘 건 존중** (오늘 사용) |

`always` vs `unless-stopped`: 내가 의도적으로 `stop`한 컨테이너를 도커 재시작 후 다시 켜느냐(`always`) 마느냐(`unless-stopped`)가 차이. 내가 끈 건 이유가 있으니 존중하는 `unless-stopped`가 무난하다.

- `docker kill` = **강제(비정상) 종료** → 재시작 정책 발동 → 부활
- `docker stop` = **정상 종료** → `unless-stopped`에선 부활 안 함

---

## 5. 자원 제한 (`--memory`, `--cpus`)

컨테이너는 기본적으로 호스트 자원을 **무제한** 사용. 한 컨테이너가 메모리를 다 먹으면 서버 전체가 마비될 수 있다. 그래서 상한을 건다.

```
--memory 1g   메모리 최대 1GB
--cpus 1      CPU 1코어 분량
```

메모리 상한 초과 시 프로세스가 강제 종료 = **OOM Kill**. `docker stats`로 실시간 사용량을 본다.

## 6. 관찰 3종 세트 — 안 열릴 때 어디를 보나

| 명령 | 확인 |
|---|---|
| `docker ps` | 컨테이너가 살아 있나? (죽은 건 `-a`) |
| `docker logs` | 앱이 뭐라고 하다 죽었나? |
| `docker inspect` | 설정(포트·재시작·자원)이 의도대로인가? |

### 컨테이너 Up ≠ 앱 UP

`docker ps`에 `Up`이어도 **컨테이너(상자)가 떴다**는 뜻일 뿐, 안의 자바 앱은 아직 부팅 중일 수 있다. 앱까지 준비됐는지는 헬스체크로 본다.

```
http://localhost:8081/actuator/health  →  {"status":"UP"}
```

## 7. 이미지 배포 — save / load

```powershell
docker save -o spring-petclinic-1.0.tar spring-petclinic:1.0   # 파일로 뽑기
docker load -i spring-petclinic-1.0.tar                        # (다른 PC) 되살리기
```

이미지를 파일 하나로 뽑아 넘기면, 받는 쪽은 소스코드도 자바 설치도 없이 그 파일만으로 같은 서버를 띄운다. → **이식성의 증명**. (Docker Hub `push`/`pull`은 선택 실습)

---

## 발표 시 강조할 3줄 요약

1. 컨테이너 = 앱 + 실행환경을 통째로 담은 상자 → 어디서든 같은 결과.
2. `-p 호스트:컨테이너`, `--restart`, `--memory/--cpus`는 각각 "연결·부활·제한"을 담당.
3. 안 열리면 `ps → logs → inspect` 순서, 그리고 컨테이너 Up과 앱 UP은 다르다.
