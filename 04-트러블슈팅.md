# 04 — 트러블슈팅

> 오류가 났을 때 여기를 본다. 증상 → 원인 → 해결 순서로 정리했다.
> **대부분의 문제는 `docker ps -a` + `docker logs`로 원인이 보인다.** 먼저 그 둘을 쳐보자.

---

## A. Docker 자체 문제

### A-1. `Cannot connect to the Docker daemon` / `docker info`가 실패

- **원인**: Docker Desktop이 안 켜져 있다.
- **해결**: Docker Desktop을 실행하고, "Engine running"/초록불을 확인한 뒤 명령을 다시 친다.

### A-2. (Apple Silicon Mac) "Rosetta installation failed" 경고

- **원인**: Docker Desktop이 x86 이미지 실행용 Rosetta를 깔려다 실패. macOS 가상화 계층 이슈.
- **해결**: 이번 실습(PetClinic, ARM 네이티브)에는 Rosetta가 **필요 없다.** 팝업에서 **`Disable Rosetta`**를 눌러 그냥 진행한다.
- 근본 해결을 원하면: 터미널에서 `softwareupdate --install-rosetta --agree-to-license` 실행 → macOS 및 Docker Desktop 최신 업데이트.

### A-3. (Windows) WSL2 관련 오류로 Docker가 안 켜짐

- **해결**: Docker Desktop 안내에 따라 WSL2를 설치/업데이트하고 재부팅한다. `wsl --update`도 도움이 된다.

## B. 이미지 빌드 문제

### B-1. `bootBuildImage`가 너무 오래 걸린다 / 멈춘 것 같다

- **원인**: 정상이다. 처음엔 베이스 이미지 등을 인터넷에서 여러 개 받아 **수 분** 걸린다.
- **해결**: 기다린다. 네트워크 로그가 계속 흐르면 진행 중인 것. **그래서 01에서 미리 빌드하라고 한 것이다.**

### B-2. `gradlew.bat`을 찾을 수 없다 / 명령이 안 먹힌다

- **원인**: spring-petclinic 폴더 **밖**에서 실행했다.
- **해결**: `cd spring-petclinic`으로 들어간 뒤 실행. 폴더 안에 `gradlew.bat`이 있는지 `dir`(macOS/Linux는 `ls`)로 확인.

### B-3. 빌드 중 자바 관련 오류 (`Unsupported class file major version` 등)

- **원인**: JDK 버전이 17 미만.
- **해결**: `java -version`으로 확인. 17 이상으로 맞춘다(1주차 환경세팅 참고).

### B-4. `docker: command not found` 또는 `gradlew`는 되는데 `docker`가 안 됨

- **원인**: Docker Desktop 미설치 또는 PATH 미반영.
- **해결**: Docker Desktop 설치 후 터미널을 **새로 열어서** 다시 시도.

## C. 컨테이너 실행 문제

### C-1. `port is already allocated` / `Bind for 0.0.0.0:8081 failed`

- **원인**: 8081 포트를 이미 다른 게 쓰고 있다(이전 컨테이너가 안 지워졌거나 다른 앱).
- **해결**:
  ```powershell
  docker ps -a                 # 8081 쓰는 옛 컨테이너 확인
  docker rm -f petclinic-app   # 있으면 강제 삭제 후 다시 run
  ```
  또는 `-p 8082:8080`으로 다른 호스트 포트를 쓴다.

### C-2. `The container name "/petclinic-app" is already in use`

- **원인**: 같은 이름의 컨테이너가 이미 있다(멈춰 있어도 이름은 점유됨).
- **해결**:
  ```powershell
  docker rm -f petclinic-app
  ```
  그다음 `run`을 다시 한다.

### C-3. 컨테이너는 `Up`인데 브라우저가 안 열린다

- **원인 1**: 앱이 아직 **부팅 중**. (컨테이너 Up ≠ 앱 UP)
  - **해결**: 20~40초 기다렸다가 `docker logs --tail 80 petclinic-app`에서 `Started PetClinicApplication` 라인을 확인. 뜨면 접속.
- **원인 2**: 접속 주소가 틀렸다.
  - **해결**: `http://localhost:8081` (컨테이너 안 8080 아님). `-p`를 `8082:8080`으로 바꿨다면 8082로.
- **원인 3**: 포트 매핑 순서를 뒤집었다(`-p 8080:8081` 같은 실수).
  - **해결**: `-p 호스트:컨테이너` = `-p 8081:8080`. 컨테이너 안 서버는 8080이다.

### C-4. 컨테이너가 떴다가 바로 죽는다 (Exited)

- **원인**: 앱이 기동 중 오류로 종료. 자원 제한을 너무 낮게 줬을 수도(예: `--memory 128m`).
- **해결**:
  ```powershell
  docker ps -a                          # Exited 확인
  docker logs petclinic-app             # 마지막 오류 메시지 확인
  ```
  메모리 부족(OOM)이면 `--memory` 값을 올려서 다시 run.

## D. 관찰/재시작 실습 문제

### D-1. `docker kill` 했는데 `docker ps`에 그대로 `Up`으로 보인다

- **원인**: 오류가 아니다. `--restart unless-stopped` 덕분에 **너무 빨리 부활**해서 그렇게 보이는 것.
- **해결**: `docker ps -a`로 상태 변화를 잡거나, `STATUS`의 "Up X seconds"가 방금으로 리셋됐는지 확인. 이게 정상 동작이다.

### D-2. `docker stop` 했는데 다시 안 살아난다

- **원인**: 정상이다. `stop`은 정상 종료라 `unless-stopped`에선 부활하지 않는다.
- **해결**: 다시 켜려면 `docker start petclinic-app`. (이게 `kill`과의 의도된 차이다)

### D-3. `docker stats`에서 못 빠져나온다

- **원인**: `stats`는 실시간 스트리밍이라 계속 화면을 갱신한다.
- **해결**: `Ctrl + C`.

### D-4. `docker inspect` 출력이 너무 길어서 뭘 봐야 할지 모르겠다

- **해결**: 필요한 값만 필터로:
  ```powershell
  docker inspect --format '{{.State.Status}}' petclinic-app
  docker inspect --format '{{.HostConfig.RestartPolicy.Name}}' petclinic-app
  docker inspect --format '{{.HostConfig.Memory}}' petclinic-app
  ```

## E. save / load 문제

### E-1. `docker load` 후 `run`했더니 이미지가 없다고 나온다

- **원인**: `save`할 때와 다른 이미지 이름/태그로 `run`하려 했다.
- **해결**: `docker load` 직후 `docker images`로 실제 로드된 이름·태그를 확인하고, 그 이름으로 `run`.

### E-2. tar 파일이 너무 크다

- **원인**: 정상이다. 이미지엔 자바 런타임까지 다 들어 있어 수백 MB다.
- **참고**: 그래서 실무에선 파일 대신 Docker Hub 같은 레지스트리(`push`/`pull`)를 더 많이 쓴다.

---

## 그래도 안 되면

1. `docker ps -a` — 컨테이너가 어떤 상태인지
2. `docker logs <이름>` — 마지막에 뭐라고 하고 죽었는지
3. 그 오류 메시지를 그대로 검색하거나 스터디 채널에 공유

> 정말 꼬였으면: 컨테이너/이미지를 다 지우고(`docker rm -f`, `docker rmi`) STEP 2부터 새로.
> PetClinic 소스가 꼬였으면 공식 리포에서 새로 clone.
